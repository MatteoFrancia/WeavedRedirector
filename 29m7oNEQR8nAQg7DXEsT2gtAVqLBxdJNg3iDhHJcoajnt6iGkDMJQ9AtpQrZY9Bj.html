VOLTA administrator guide outline

 

  • VOLTA Users

 

      • VOLTA defines different kind of platform’s users
      • each one of them is characterized by a different access level to the application
      • the access level defines the operations that a user can perform and the sections he can enter
      • Today we are going to focus on the VOLTA administrator user
      • VOLTA Administrator
        • unique → there’s only one admin per installation
        • the only user who can log on a fresh installation where the license agreement hasn’t yet been accepted
        • can’t define optimization problems
        • can’t run simulations
      • VOLTA full user
        • as a full user can perform all the operation permitted by VOLTA
        • items / elements
      • VOLTA Light user
        • to access VOLTA environment and see simulation results
        • without viewing all the other useless (for their needs) options
        • this kind of user were created to allow managers
      • VOLTA Simulation user
        • kind of user focused on the simulation execution
      • VOLTA Data user
        • Add Data and information to the application creating and modifying elements

  • Queues

 

      • VOLTA queues are temporary containers for execution requests
        • these requests are JMS messages generated by the VOLTA application when a user submit an evaluation
      • As Matteo told you yesterday there is only a single JMS queue defined in the A.S. resources through the VOLTA installer
        • and the application itself manages them
        • all the queues that an Admin can define inside the application are virtual queues
      • [SLIDE] This slide simplifies how the queues and evaluators interaction works
        • as shown, once a message is placed in a queue a VOLTA evaluator attached to the queue can take the message and execute it
        • the message contains all the information the evaluator needs to perform the optimization
        • during
      • As you can see
        • different users can submit requests to the same queues
        • different evaluators can listen to the same queues
        • each evaluator can listen only one queue
      • Keep in mind that all the nodes requested by an evaluation have to be enabled on the evaluator in order to let it take the jms message
        • mF has to be installed on the same machine as the evaluator with a valid license
        • all the necessary third party sw have to be installed with all the required licenses
      • using queues the administrator can assign computing resources to users and group
        • assigning them to a queue → to assign users and group to a queue…
      • Clicking on the queue name you can enter the queue administration dashboard where you can see
        • Request ID → Session ID for .prj-based sessions;
          • corresponds to Design ID for .prc-based sessions.
        • Session ID: the ID number assigned to that session by  VOLTA.
        • Local ID: only for .prc sessions. It corresponds to the incremental Design ID within the .prc session.
        • this defines which workflows can be executed on this queue and which can not
        • general information
        • a chart showing the number of messages put in the que per day
        • a list of the pending messages placed in the queue with some useful information
        • if there are evaluators attached to the queue we can see the list of all the enabled node on these evaluators
